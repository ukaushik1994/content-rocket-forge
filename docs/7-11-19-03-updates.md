## CRITICAL: Shared Conversation RLS Leaks All Shared Data

**RLS policy on `ai_conversations`:**
```
"Shared conversations accessible by token" → SELECT → (is_shared = true AND share_token IS NOT NULL)
```
**Roles:** `anon, authenticated`

This means ANY anonymous or authenticated user can `SELECT * FROM ai_conversations WHERE is_shared = true` and see EVERY shared conversation in the system — titles, timestamps, user IDs. The frontend filters by `share_token` but the DB doesn't enforce it.

**Fix for Lovable:** The policy should be dropped and replaced with a function-based policy or the app should use a Supabase RPC for shared conversation lookups:

```sql
-- Drop the broad policy
DROP POLICY "Shared conversations accessible by token" ON public.ai_conversations;

-- Replace with one that requires the token to be provided in the request
-- Option: use a function that takes share_token as parameter
CREATE OR REPLACE FUNCTION public.get_shared_conversation(p_token text)
RETURNS SETOF ai_conversations AS $$
  SELECT * FROM ai_conversations WHERE share_token = p_token AND is_shared = true;
$$ LANGUAGE sql SECURITY DEFINER;
```

Then update `SharedConversation.tsx` to use `supabase.rpc('get_shared_conversation', { p_token: shareToken })` instead of direct `.from().select().eq()`.
